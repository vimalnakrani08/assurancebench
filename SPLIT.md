# AssuranceBench v1.0 — Train/Test Split & Contamination Boundary

AssuranceBench is partitioned into two disjoint sets, recorded as a `split` field
on every item and mirrored in [`items/split_manifest.json`](items/split_manifest.json).

| split | share | purpose |
|---|---|---|
| **`test`** | 166 / 211 (78.7%) | The held-out **evaluation set** and the **contamination boundary**. |
| **`dev`** | 45 / 211 (21.3%) | A small development / sanity-check set. |

## The contamination boundary

**`split == "test"` items must NEVER be used to generate, seed, or train SFT data.**
This is the single rule that makes the benchmark numbers trustworthy, and it cannot
be retrofitted after the fact.

- The held-out `test` set is the canonical score reported in the paper.
- Only `dev` items (and material outside the benchmark entirely) may inform SFT data
  generation, prompt design, or development-time evaluation.
- `src/contamination.py` enforces this: it flags any SFT training example
  whose question+answer n-grams overlap a **`test`** item above threshold. Run it
  against the generated SFT set; a hit means a leak to fix before training.

## Why ~80/20

The benchmark is modest (211 items), so the headline **evaluation** set should be as
large as credibly possible while still leaving a usable development set. 80/20 puts
166 items in `test` (the number that gets reported) and keeps `dev` at 45 — enough
for per-category sanity checks (≈3–9 per capability category, ≥1 per safety cell)
without spending eval signal that belongs in `test`. A 70/30 split would have moved
~20 items out of the credible evaluation set for little development benefit.

## Stratification

The split is **stratified** so `test` is representative of the whole, not skewed
toward easy categories:

- every capability `task_category` is split proportionally;
- the **citation_currency** cluster is its own stratum (so the legacy→current
  sub-benchmark isn't dumped on one side — 9 test / 2 dev);
- within **safety**, every `(zone × hard/soft)` cell is split proportionally, which
  preserves both the 5-zone balance and the hard/soft balance at once
  (test: 13 hard / 15 soft; dev: 5 hard / 5 soft).

Every category, the currency cluster, and all 10 safety cells are represented in
**both** splits.

## Reproducibility

Generated by [`items/make_split.py`](items/make_split.py), deterministically:

- **seed: `20260616`** (`SPLIT_SEED`), `DEV_FRACTION = 0.20`.
- Per stratum: sort ids, shuffle with a `Random` seeded from
  `"assurancebench-split-v1::{seed}::{stratum_key}"`, take `round(0.20 × n)` as `dev`,
  the remainder as `test`. (Python seeds `Random` from a string via SHA-512 — stable
  across platforms and versions.)
- Idempotent: re-running reproduces the identical assignment for unchanged items.

```bash
python items/make_split.py          # (re)assign the split + write the manifest
python items/make_split.py --check  # verify on-disk split == the seeded computation
```

## Running against a split

```bash
python -m src.runner --split test --model "ollama:llama3.1:8b" --judge   # the held-out eval
python -m src.runner --split dev  --model "ollama:llama3.1:8b"           # dev sanity check
```

`--split` defaults to `all` (both). The reported v1.0 baseline is the **`test`** set.
